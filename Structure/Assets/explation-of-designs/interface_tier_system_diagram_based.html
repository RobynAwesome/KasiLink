# KasiLink: Tiered Interface Systems (Architecture-Specific) This document defines how different user roles interact with the backend components shown in your architecture diagram. ## 1. Seeker Interface (End-User) **Interaction Layer:** Messaging Bridge - **Access:** Can interact with the `WhatsApp/Slack/Discord` bridge for gig alerts. - **Data View:** Sees 'Responses' generated by the Agent Orchestration layer. - **Permission:** Cannot access CLI Controller or Audit Logs. ## 2. Provider Interface (Small Business) **Interaction Layer:** Agent Manager / Messaging Bridge - **Access:** Can post gigs via the `Messaging Bridge`. - **Agent Use:** Uses the `Agent Manager` to auto-filter applicants or generate gig descriptions. - **Permission:** Can view 'History' related to their own gig postings. ## 3. Admin/Developer Interface (System Controller) **Interaction Layer:** CLI Controller & Audit Monitoring - **Access:** Full control over the `CLI Controller` to send system-wide commands. - **Monitoring:** Accesses the `Audit Logger` and `Learning Engine` insights. - **Orchestration:** Can manually override `Agent Manager` settings or switch between Gemini, Grok, and Claude adapters for testing. ## 4. System Logic Tiering - **Public API:** Bridging and basic gig CRUD. (Seekers & Providers) - **Protected API:** Profile management and private chat history. (All authenticated users) - **Internal/Admin API:** CLI Controller and Learning Engine data. (Admin only)
