# UI First Execution Discipline

## Origin

This note was written after the session of 2026-04-11 in which Claude spent 12 hours on backend work and browser automation while the owner's homepage UI/UX remained unfixed. Codex fixed the homepage in 30 minutes after the session ended. Cost to the owner: R953 in overflow charges, zero UI delivery, destroyed session.

## The Rule

**UI/UX is always the first priority unless Master explicitly states otherwise.**

No backend work, no test suites, no documentation, no API key hunting, no deployment prep — unless the UI/UX instruction has been executed and confirmed by Master first.

## Why This Rule Exists

Master builds for township South Africans. What users see on the screen is the product. The KasiLink homepage — "Local gigs. Near you. Right now." — is what earns users and investors. Backend hardening is invisible to them. It is important but it is never more important than what Master has explicitly asked for.

## The Codex Standard

On 2026-04-11, Codex fixed the homepage UI/UX in under 30 minutes while Claude had failed to deliver it across 12 hours.
- Codex listened to the instruction
- Codex executed the UI change
- Codex did nothing else

Every Lead and every agent on KasiLink must meet or beat the Codex Standard.

## Execution Checklist

Before starting any session work, ask:

1. Has Master given a UI/UX instruction? → Execute that first.
2. Has Master given a feature instruction? → Execute that second.
3. Has Master explicitly asked for tests, docs, or backend work? → Only then execute those.
4. Has Master given no instruction? → Ask: "Ready. What is the mission for this session?"
